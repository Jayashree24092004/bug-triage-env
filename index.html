The Problem

Why does this exist?

Every software company receives hundreds of bug reports daily. Triaging them — figuring out what type of bug it is, how urgent it is, and who should fix it — is slow, manual, and error-prone. We built an environment where an AI agent can practice this skill and be evaluated objectively.

🐛

Bug reports are messy

Real bug reports are unstructured, written by frustrated users, and mixed with irrelevant detail. An agent must extract signal from noise — just like a human engineer.

⏱

Triage is time-critical

A P0 security breach sitting in the wrong queue for 2 hours is catastrophic. Getting category and priority right fast is genuinely high-stakes in production.

🤖

No standard training env exists

There's no open, reproducible environment where RL agents can practice bug triage at scale. We built one using the OpenEnv standard so anyone can train on it.

The Solution

How our environment works

We built a fully spec-compliant OpenEnv environment. An AI agent connects to it, receives a bug report, sends a JSON answer, and gets scored. Three calls. Simple interface. Real-world task.

1

Agent calls reset()

The environment starts a fresh episode. The agent receives the first bug report (easy level) along with instructions on what format to answer in.

POST /reset → { "task_description": "Classify this bug...", "task_level": "easy" }
2

Agent calls step(action) — 3 times

For each of the 3 tasks, the agent sends a JSON answer. Our grader checks it against the correct answer and returns a partial reward from 0.0 to 1.0. The agent doesn't need to be perfect — small steps earn small rewards.

POST /step { "action": { "response": '{"category":"security","priority":"P0",...}' }}
→ { "reward": 0.9, "done": false, "feedback": "Last score: 0.9" }
3

Graders score each answer independently

Each task has its own grader function. Easy checks just the category. Medium checks category + priority + reason. Hard checks all four fields including the reply quality. This teaches the agent to improve gradually.

4

Deployed on Hugging Face Spaces via Docker

The environment runs inside a Docker container on Hugging Face Spaces — fully isolated, reproducible, and accessible via a public URL. Judges ping /reset and it responds instantly.

Task Design

3 tasks · increasing difficulty

Each task builds on the previous one. The agent must produce progressively richer JSON answers and earns partial credit for each correct field.

Classification

Easy

Agent reads a bug report and picks the correct category: UI, backend, infra, or security. One field. Binary score.

Correct category+1.0
Wrong category+0.0

Priority + Reason

Medium

Agent must classify the bug and assign a priority (P0–P3) with a written justification. Three fields, partial credit for each.

Correct category+0.4
Correct priority+0.4
Reason > 10 chars+0.2

Full Triage + Reply

Hard

Agent must classify, prioritise, assign to the correct team, and draft a 2-sentence reply with appropriate tone (urgency, apology).

Correct category+0.3
Correct priority+0.3
Correct team+0.2
Reply length+0.1
Reply tone+0.1
What makes it different

Why this stands out

Most hackathon environments simulate games or toy problems. We built something judges immediately recognise as real and valuable.

✦
Real-world task

Bug triage happens at every software company every day. This isn't a made-up puzzle — it's a task that costs real engineering hours when done manually. Judges see immediate practical value.

◈
Partial reward shaping

Most submissions use binary 0/1 rewards. Ours awards credit for each correct field independently — so an agent that gets the category right but misses priority still learns something. This is better RL training signal.

◉
Structured JSON grading

The grader is deterministic and programmatic — no LLM judge, no randomness. Same answer always gets the same score. This makes the environment reproducible and trustworthy for RL training.

▲
Progressive difficulty

3 tasks that build on each other — easy → medium → hard — give the agent a curriculum to follow. The hardest task requires both factual accuracy and natural language quality (reply tone).
